CI/CD Pipelines

Introduction

Over recent years, the software development industry has increasingly shifted focus to automated and continuous code deployment and open communication between developers and operations managers. Software development companies who want to keep a competitive foothold in today's market should seriously assess the benefits of applying these methods, namely Continuous Integration/Continuous Deployment (CI/CD) pipelines and a DevOps culture.

The Workflow

Development

Automated code deployment and testing has increased productivity of software development in the industry immensely. In a CI/CD pipeline, after a developer plans and writes his or her code, they push it to a shared repository. Automation integrates the developer's code into the central code base, builds it, then begins testing.

Testing

When testing, the code is run through both functional and non-functional testing methodologies.

Functional tests are those that assess the code's abilities to fulfill the business requirements (i.e., the criteria and use cases laid out by the dev team and/or stakeholders). Functional tests can be broken into the following categories:

  • Unit testing - Test the code in individual pieces (i.e., each class and/or function).
  • Integration testing - Test the code's functionality when joined with its dependencies, such as databases and libraries that the code accesses.
  • System testing - An end-to-end “black box” test, typically conducted by a different team than the developers, to ensure the code fulfills the business requirements expected and delivers proper results.
  • Acceptance testing - The code works the way the end user expects it to work.

Non-functional testing focuses on the high-level performance and operations of the code, abstracted from the business-level criteria laid out by the dev team and/or stakeholders. Examples of non-functional testing include:

Performance testing
  • Load testing - Emulate incrementally higher amounts of demand on the running code to confirm ability to perform at predicted usage amounts.
  • Stress testing - Place estimated peak load (and beyond) on the running code to detect failure points.
  • Endurance testing - Place sustained demand on the running code to assess performance with consistent load. Key in discovering memory leaks.
  • Spike testing - Emulate drastic immediate increases in application usage and assess ability to sustain functionality.
  • Security testing
  • Usability testing
  • Compatibility testing
