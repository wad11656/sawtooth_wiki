Azure Tutorial Part 2: Write the Code

1. Create a Blank Visual Studio Solution

  1. Set up Visual Studio:
    1. Download the Visual Studio installer for your platform.
      1. For Windows, choose the Community edition.
    2. When prompted by the Visual Studio installer, select a .NET Core workload:

    3. Launch Visual Studio.
  2. On the launch screen, select Create a new project (Windows) or + New (Mac).
  3. Search for & click Blank Solution.
  4. For the Location, to the directory above your repo.
  5. Match the Solution/Project Name with the name of your repo so the solution folder merges with it.
  6. Click .
    1. If prompted, click to merge the solution and repo folders.

Repos <-- Use as the solution location
  HelloWorld <-- Use as the solution name
    .vs <-- Gets created alongside the Solution (You can ignore it)
    .git
    README.md
    HelloWorld.sln <-- Solution gets created here

2. Create the Main C# App

  1. Right-click the solution in the Solution Explorer and click Add New Project…
  2. Search for & click C# Console App (.NET Core).
  3. We'll Name the C# console app's project MainApp.
  4. Make sure the Location is the repo's folder.
  5. Click .
  6. There should be a file at HelloWorld > MainApp > Program.cs that contains the following line: 
    Console.WriteLine("Hello World!");

3. Create the Test App

  1. Right-click the solution in the Solution Explorer and click Add New Project…
  2. Search for & click C# MSTest Test Project (.NET Core).
  3. We'll Name the Test Project TestCases.
  4. Make sure the Location is the repo's folder.
  5. Click .
  6. There should be a file at HelloWorld > TestCases > UnitTest1.cs that contains skeleton code for test cases.

4. Confirm the Repo's Folder Structure

  1. Navigate to your local repo (or look in Visual Studio's Solution Explorer) and confirm your file structure looks like the following:
    Repos
  HelloWorld
    .vs
    .git
    MainApp
      obj
      bin
      MainApp.csproj
      Program.cs
    TestCases
      obj
      bin
      TestCases.csproj
      UnitTest1.cs
    README.md
    HelloWorld.sln

How Will the App Work?

  1. The Azure DevOps pipeline will save MainApp's output (Hello World!) to a .txt file.
  2. Our TestCases test project will read the .txt file.
  3. The test project will then validate the contents of the .txt file to be Hello World! by running the following 2 tests:
    1. Confirm the 4th char in the .txt file is l (Hello World!)
    2. Confirm the 5th char in the .txt file is o (Hello World!)

5. Rename & Prep the Test App

  1. In the Solution Explorer, right-click and rename UnitTest1.cs → CharN.cs.
    • Note: The name CharN is representative of the fact that each test will be validating the char at some Nth position in the .txt file.
  2. If prompted, click to perform a rename of all instances of UnitTest1.
    1. Otherwise, in CharN.cs, change public class UnitTest1 → public class CharN.
  3. Change the first [TestMethod] from public void TestMethod1() → public void Char4().
    • Note: This method will validate the 4th char in our .txt.
  4. Add using System.IO; at the top of CharN.cs.
    • Note: Using this library will allow reading a local .txt file.

6. Write the Test App's Code

  1. Add the following directly above your [TestMethod]:
    string txtOutput = File.ReadAllText("WriteLineOutput.txt");
string letter = "";
    • Note: These variables are accessible to all [TestMethod]s defined within the same class.
  1. Add the following code under a [TestMethod] for each char in the .txt you want to validate:
    [TestMethod]
  public void Char<pos of char to validate>()
    {
     letter = txtOutput.Substring(<pos of char to validate, where 1st char is 0>, 1);
      Assert.AreEqual(letter, "<expected letter>");
    }
    • Note: The 1 in (<pos of char to validate, where 1st char is 0>, 1) is the char length of the substring. Since we're validating just 1 letter, of course our substring is 1 char long.

In our example, we'll validate the 4th and 5th chars. So, CharN.cs should look like this:

CharN.cs 

using Microsoft.VisualStudio.TestTools.UnitTesting;
using System.IO;

namespace UnitTestProject1
{
    [TestClass]
    public class CharN
    {
        string txtOutput = File.ReadAllText("WriteLineOutput.txt");
        string letter = "";

        [TestMethod]
        public void Char4()
        {
            letter = txtOutput.Substring(3, 1);
            Assert.AreEqual(letter, "l");
        }
        [TestMethod]
        public void Char5()
        {
            letter = txtOutput.Substring(4, 1);
            Assert.AreEqual(letter, "o");
        }
    }
}

7. Create a .txt Locally for Testing

We are now going to create a WriteLineOutput.txt locally so we can make sure our tests work locally before uploading our project to our remote repo. However, we will not upload this local WriteLineOutput.txt to our remote repo because our pipeline is going to create its own .txt files each time it runs instead.

In order for CharN.cs' string txtOutput = File.ReadAllText(“WriteLineOutput.txt”); variable declaration to work, we'll need to put WriteLineOutput.txt right alongside the TestCases project's TestCases.dll test container. The .dll test container is located in Repos > HelloWorld > TestCases > bin > Debug > netcoreappX.X > TestCases.dll.

  1. Create a file called WriteLineOutput.txt with the exact same contents of your MainApp's Console.WriteLine() function (Default=Hello World!).
  2. Place the .txt in Repos > HelloWorld > TestCases > bin > Debug > netcoreappX.X:
    Repos
  HelloWorld
    TestCases
      bin
        Debug
          netcoreappX.X
            TestCases.dll <-- You won't see this until you run your tests in Step 8.
            WriteLineOutput.txt

8. Run Your Tests

  1. Click > .

Your tests should pass, as seen in the Test Explorer:

Now let's make Char5 fail.

  1. In the Char5 [TestMethod], change the <expected letter> from “o” to something else (i.e., “l”):
    [TestMethod]
  public void Char5()
    {
     letter = txtOutput.Substring(4, 1);
     Assert.AreEqual(letter, "l");
    }
  2. Click > .

Char5 should fail, as seen in the Test Explorer:

Note the helpful Message: Assert.AreEqual failed. Expected:<o>. Actual:<l>.

9. Create the .gitignore and Push to the Remote Repo

  1. In the root folder of your repo (HelloWorld), create a .gitignore file.
  2. Add the following contents to the file:
    .vs/
**/bin/
**/obj/
TestResults/
    With these .gitignore file/folder exceptions, the following files will be uploaded:
    Repos
  HelloWorld
  ❌.vs
  ❌.git
    .gitignore
    MainApp
    ❌obj
    ❌bin
      MainApp.csproj
      Program.cs
    TestCases
    ❌obj
    ❌bin
      TestCases.csproj
      CharN.cs
  ❌TestResults
    README.md
    HelloWorld.sln
  3. In the Char5 [TestMethod], change the <expected letter> back to the correct letter (“o”):
    [TestMethod]
  public void Char5()
    {
     letter = txtOutput.Substring(4, 1);
     Assert.AreEqual(letter, "o");
    }
  4. In a terminal within the repo's root, run git add ., git commit -m “your_comment_here”, and git push.
